# DE-KEA
_IKEA? I hardly know her!_

Get rid of the IKEA in your life. DE-KEA.

## User journey
- On the homepage you see a demo video start playing immediately.
  - There's a two-line explanation of the product and a CTA.
- Click the 'Take photo' button
- Take a photo of a room (we're encouraged to do it in landscape).
- Now we go to a half and half page layout.
  - The top half has the current image, the bottom half is the conversation.
  - Each pane can be expanded to full page or re-
  - Above the top half is a thin band with 'DE-KEA: Interior design without IKEA.'
- A new image is returned to the image pane. All the IKEA objects hav been removed from it.
- The conversation pane includes the returned image from the agent, followed by rich rendered Markdown which lists the identified IKEA objects along with witheringly critical comments about each one.
  - Note that the recycling centre line would need web browsing.
```markdown
## Removed:
1. A tacky red lamp
2. Four astonishingly gaudy prints on the right hand wall
3. A BILLY bookcase (yawn)
<!--v1.4 Pi required for the web search/browsing subagent. We'll hardcode it in for v1.1 but remove for v1.2 and v1.3.-->

Then we can start designing a room that is tasteful, a room that you're proud of.
```

- The user responds. They say they'd also like to replace the sofa. Another image is returned in the conversation, this time without the sofa as well.
- We now go through the items. They're asked what style of lamp they'd like -- modern, retro, classic? They can try a few options. They see how a new object would look in the room.
- (v1.4) Each time the user is happy with a couple of options for the room, they are told where they can get the new object from. For example there's a British Heart Foundation down the road with good quality sofas.

    - eBay, Gumtree, Facebook marketplace

## Claude spec

This is the build blueprint I (Claude Code) follow to implement de-kea. It is deliberately scoped to **v1.1 = the hardcoded walkthrough (TODO #1)**.

### v1.1 goal & guiding constraint
- Ship the complete UX shell as a scripted walkthrough of the user journey.
- **v1.1 hits nothing.** No network calls, no ElevenLabs, no OpenRouter, no image models. Every image and every chat response is canned content committed to the repo. The app must run fully offline.
- **The user drives the journey with real free-text input** — they type into the conversation as they would in the real product. Only the *responses* are canned: user input advances the scripted state machine, but the agent's replies (images + Markdown) are pre-authored fixtures, not generated.
- The point of v1.1 is to lock the look, feel, animations, and journey flow so later phases only swap the canned responses for live model calls behind the same interfaces.

### Tech stack (v1.1)
- **SvelteKit** app.
- **Tailwind CSS** for styling and the design system.
- **Vitest** (+ `@testing-library/svelte`) for unit/component tests; **Playwright** for the E2E journey. Red/green TDD, pyramid shape (many unit, fewer component, a couple of E2E).
- **Cloudflare Pages** via `@sveltejs/adapter-cloudflare`, deployed and managed through the Wrangler CLI from the start.
- No backend, no database, no auth. Static assets only.

### User journey as a state machine
The journey is a scripted state machine. The user advances it with **real free-text input** typed into the conversation (plus the homepage CTA). Input maps to the next scripted step; the reply for each step is canned. Optional suggested-reply chips can seed common inputs, but typing is the primary path.

1. `home` — demo video autoplays; two-line pitch + "Take photo" CTA.
2. `capture` — "Take photo" is **fully faked**: no camera, clicking it advances straight to the workspace with a canned "before" image.
3. `zapped` — workspace shows the canned "after" image (IKEA removed) in the image pane; conversation pane renders the canned removal list (rich Markdown, witheringly critical) + the follow-up prompt.
4. `replace-sofa` — user types a request to also replace the sofa; a second canned image (no sofa) appears in the conversation and updates the image pane.
5. `style-lamp` — user is offered lamp styles (modern / retro / classic); each choice swaps to a canned in-room render so they can "try" options.
6. `end` — journey wraps (sourcing / recycling copy is stubbed; the web-search dependent lines are hardcoded per the v1.1 note and flagged as TODO for v1.4).

Because input is free text but responses are canned, each step matches the user's message loosely (keyword/intent heuristics) to decide the transition, then returns the pre-authored response for that step. State lives client-side (a Svelte store / XState-style machine). Each transition is deterministic and independently testable.

### Routes & components
- `/` — homepage (`Hero`, `DemoVideo`, `Cta`).
- `/workspace` — the half-and-half layout.
  - `TitleBand` — thin band: "DE-KEA: Interior design without IKEA."
  - `ImagePane` (top) — current image; **expandable to full page**.
  - `ConversationPane` (bottom) — message list of images + rendered Markdown; **expandable to full page**.
  - `ChatInput` — free-text input the user types into to advance the journey.
  - `SuggestedReplies` — optional chips that seed common inputs into `ChatInput`.
  - `PaneToggle` — handles the expand/collapse of either pane.
- `lib/journey/` — the state machine + step definitions.
- `lib/fixtures/` — canned images (`static/`), Markdown copy, and the scripted conversation script (one data file that drives the whole walkthrough).
- Markdown is rendered to sanitised HTML at build time or via a small runtime renderer.

### Design system (modern art gallery)
- Hard edges, sharp corners, black & white, judicious bold splashes of colour. References: frieze.com, designmuseum.org.
- Tailwind theme tokens: mono/near-black + paper-white base, one or two accent colours used sparingly.
- Every state transition has an intentional animation/transition (pane expand, image swap, message enter). Prefer Svelte transitions / crossfade.
- Halftoning as a texture/treatment motif (line 46 of the journey is incomplete in the source; treat halftone as a decorative image/section treatment).

### Testing strategy
- **Unit:** the journey state machine — every transition, guard, and terminal state.
- **Component:** panes render the right fixture for a given state; expand/collapse toggles; typed input advances state and returns the canned reply; the input→step matching heuristic; Markdown renders and is sanitised.
- **E2E (Playwright):** one happy-path run through `home → end`; one run exercising pane expand/collapse.
- TDD: write the failing test for each transition/component before implementing it.

### Deployment
